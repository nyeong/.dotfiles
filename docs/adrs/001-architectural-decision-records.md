# Architectural Decision Records

## 상황

규칙을 기록해야할 필요성이 있음

- 다루는 호스트가 많아지니 지켜야할 규칙들이 생김
- 규칙을 nix로 강제하는게 제일 좋겠지만, 혼자 작업하는 레포에는 오버킬임
  - 그냥 머릿속의 규칙을 알아서 준수하고 바로 짜고 빌드해서 결과물을 보고 싶음
- 근데 나도 규칙 까먹을 때가 많음
- 종종 LLM에게 질의하는데 맥락 던져주기 귀찮음

## 선택지 및 결정

1. GitHub Issue에 기록하기
2. 별도의 장소에 문서로 남기기 (org)
3. 레포 내에 문서로 남기기

(3)을 선택하기로 함.

- 작업소요를 줄이는게 중요하므로 웹 접속이 필요한 (1)은 제외.
- (2)도 유의미하나, LLM에게 맥락 던져주기 힘듦으로 제외.

## 세부 내용

- `/docs/adr/{number}-{slug}.md`로 규칙에 대한 의사결정을 남기자.
- 각 문서는 아래의 내용이 들어가야함
  1. 상황
  2. 선택지 및 결정
  3. 세부 내용
  4. 기대하는 결과
- 간결하게

## 기대하는 결과

- "아 이거 어떻게 하려고 했더라" 생각 들 때 문서 보면 된다
- 생각 정리 자체를 문서쓰면서 하면 된다
- 나중에 문서 많이 쌓이면 찾기 어려울텐데, 많이 쌓이면 그때 고민하자
